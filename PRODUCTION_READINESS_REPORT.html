What's actually ready

This is the part worth being confident about — it was rebuilt this session against real recorded audio, not just unit-tested in the abstract.

  • Speech detection is now acoustically grounded. Silero VAD replaced WebRTC VAD as the segmentation gate, validated against a real recorded meeting: known-speech windows scored 0.7–1.0 confidence, silence and room noise scored near-zero, with zero false segments across 60s of pure silence and 60s of realistic room tone.
  • Echo/leakage suppression is real, not cosmetic. A correlation-based ducking gate was added to the existing WebRTC AEC wrapper, specifically to catch loud residual echo (mic picking up speaker output with no headphones) that the old ratio-based ducking couldn't touch above 0.05 RMS. Verified against the actual reported failure mode (a YouTube clip played through speakers, replayed 3×, zero mic speech) — the fix took it from consistent hallucinated "Speaker 1" text down to zero false segments in the confirmed scenario.
  • Backend has real resilience primitives. A recent hardening pass added a circuit breaker around STT calls, a thread supervisor that detects hung/dead pipeline threads and auto-restarts them (capped retries), bounded queues with drop-oldest backpressure instead of unbounded memory growth, device-fallback when a configured mic/speaker disappears, and crash/auto-recovery for an in-progress meeting via a periodically-flushed `active_meeting.json`.
  • The STT engine abstraction is clean. Three backends (local ONNX/DirectML, local CTranslate2, remote HTTP) sit behind one `STTEngine` interface with a single `transcribe()` method — swapping or adding an engine doesn't touch VAD/pipeline code.
  • Tauri's permission surface is tightly scoped. The desktop shell only requests window-management capabilities; no filesystem, shell-execute, or HTTP permissions beyond spawning its own backend sidecar in Rust.

Deep dive: the audio→text pipeline, and how production systems actually do it

This is the part you asked to look at specifically. Three layers — capture, VAD/segmentation, and STT — each compared against how a production voice pipeline (Otter/Fireflies-style meeting bots, or a raw provider like Deepgram/AssemblyAI) handles the same problem.

Audio capture

Two independent soundcard-based threads (WASAPI on Windows) poll a mic and a speaker-loopback device in fixed 30–60ms blocks, pushed onto queues. A recent patch added exception-wrapped device fallback (falls back to system default if a named device disappears) and flatline detection (re-opens the WASAPI recorder after ~3s of zero-variance audio, a real and common failure mode for USB devices on sleep/wake). That's a genuinely production-relevant fix, not cosmetic.

What's missing next to a production capture layer: no sample-rate/format negotiation fallback if the requested 16kHz isn't natively supported (WASAPI shared mode usually resamples transparently, but this is unverified across devices), no jitter buffer or explicit under/overrun counters exposed anywhere, and capture threads use blocking rec.record() calls rather than a callback/lock-free ring buffer — acceptable at this data rate on a desktop CPU, but not how low-latency audio I/O is normally structured (JUCE, PortAudio, and WASAPI event-driven callbacks all avoid blocking calls on the audio thread specifically to bound worst-case latency).

Voice activity detection & segmentation

Silero VAD (ONNX, CPU) replaced WebRTC VAD this session — the single highest-leverage fix in the whole pipeline, since WebRTC VAD's ~50% true-positive rate at a usable false-positive rate was letting non-speech blips through as "segments" that downstream STT engines would then hallucinate real-sounding text onto. Segmentation itself uses a fixed silence-hangover threshold (mute for 500ms → cut the segment) with pre-roll padding to avoid clipping word onsets, plus a per-segment confidence floor (mean Silero probability) and a minimum-voiced-duration floor, both enforced in one place instead of two independent, driftable thresholds.

Why partials matter more than they sound like they do Every production competitor (Deepgram, AssemblyAI, Google STT, Zoom, Teams) streams interim words within a few hundred milliseconds and finalizes them later. This app shows nothing until the full 500ms-hangover-plus-inference latency has elapsed per utterance — noticeably "laggier" than any commercial competitor, independent of raw model accuracy.

Speech-to-text engines

Three interchangeable backends: Moonshine (ONNX + DirectML, the default — fast, but ships with literally no confidence output, which is why the VAD-side confidence gate above is load-bearing for it specifically), faster-whisper/CTranslate2 (CPU or CUDA, has real no_speech_prob/avg_logprob signals, hardened this session with condition_on_previous_text=False and an explicit compression-ratio guard against repetition loops), and a remote HTTP engine that posts WAV segments to a GPU box over Tailscale. All three are called as one blocking, synchronous request per finished utterance.

TransportOne-shot HTTP POST per utterance (remote engine)Persistent WebSocket for the whole session — provider docs explicitly recommend not opening/closing per utterance; 20ms chunk streaming is the 2026 norm

DiarizationHardware channel split (mic = "me", loopback = "them") — caps at exactly 2 sidesNeural embedding-based diarization (pyannote-style clustering) on a single mixed stream — handles N speakers, at the cost of needing a real model and being wrong more often on overlapping speech
Architectural ceiling, not a bug The 2-channel-hardware approach to diarization is a legitimate, low-cost design choice — it's why this app never needs a heavy neural diarization model and gets clean "me vs. everyone else" separation almost for free. But it is a ceiling, not a tuning problem: any call with 3+ real participants on either side of the split (a group meeting, a panel) collapses into one undifferentiated "Speaker 2." Every production competitor solves N-speaker diarization because that's the common case for "meeting," not the edge case.

The single global stt_lock is worth calling out on its own: it exists to prevent concurrent-execution crashes in the DirectML/ONNX runtime, which is a real constraint for Moonshine specifically — but it means cross-talk (both people speaking near-simultaneously, extremely common in real conversation) forces one channel's transcript to wait behind the other's inference time, every time. Production ASR serving layers solve this with per-stream model instances or dynamic batching precisely so concurrent streams don't serialize.

Everything else: gaps by category

Ordered roughly by blast radius. "Critical" means it either breaks the product outright or creates real legal/financial exposure; "major" is a genuine production blocker; "minor" is real but not urgent.

Security

CriticalHugging Face token committed to git history

A real API token is hardcoded in remote_stt_server.py and was tracked in backend/.env. Removing it from the working tree doesn't remove it from history — it needs rotating on huggingface.co and purging from history if the repo is ever made public or shared.

CriticalNo authentication on any API surface

The FastAPI backend and the remote GPU STT server both accept requests from anyone who can reach them, with CORS wildcarded and credentials allowed — the remote server binds 0.0.0.0:8001, reachable to the whole Tailscale network with zero auth.

MajorTauri Content-Security-Policy is disabled

tauri.conf.json sets csp: null — no restriction on script/style/connect sources in the webview. Low practical exposure today (no dangerouslySetInnerHTML found), but it removes a defense-in-depth layer against any future dependency compromise.

MajorNo encryption at rest

Saved transcripts and audio live as plaintext JSON and WAV in a local folder that has already grown to 328MB across 58 files with no rotation, archiving, or cleanup mechanism.

Legal & compliance

CriticalNo recording consent or notification mechanism

The app silently records both mic and system audio with no on-screen "recording" indicator visible to other call participants and no consent prompt. 11+ US states (California, Florida, Illinois, Washington among them) require all-party consent for recording private conversations, with penalties up to $2,500 per occurrence and potential criminal exposure — this is a real liability the moment it's used on a call with anyone outside the operator.

MajorREADME makes a privacy claim the code contradicts

"No audio data ever leaves your machine" is stated as a core feature, but a fully-implemented remote engine posts raw audio segments to an external GPU server whenever that model is selected — the claim is factually false for a real, shipped code path, not a hypothetical one.

MinorNo data retention or deletion policy

No bulk export, no automatic expiry, no documented way to purge stored recordings — table stakes for any tool handling other people's speech.

Testing & CI

CriticalZero automated tests anywhere

No pytest files, no vitest/jest config, nothing beyond third-party library tests in venv/. Every fix this session was validated by hand against real audio in ad-hoc scripts — that doesn't survive the next refactor.

CriticalNo CI configuration exists

No .github/workflows/ or equivalent. Nothing runs on push or PR; there is no automated gate between a commit and a release build.

Packaging & release

CriticalThe Tauri sidecar binary is a 0-byte placeholder

src-tauri/bin/nexus-ai-backend-x86_64-pc-windows-msvc.exe — the file the release build is configured to bundle as the backend — is empty. A tauri build run right now ships a non-functional app.

MajorNo scripted, reproducible build process

backend/dist/ has 4 differently-named exe artifacts and backend/build/ has 3 differently-named subfolders (~195–200MB each) — strong evidence of manual, ad-hoc PyInstaller invocations rather than a single documented build script. No .spec file exists anywhere.

MajorNo auto-update mechanism

No tauri-plugin-updater, no update config. Once installed, there is no way to push a security or bug fix without a manual reinstall.

MinorUnpinned Python dependencies

requirements.txt specifies lower bounds only (faster-whisper>=1.0.0, etc.) — a build today and a build in six months can silently pull different versions.

Observability & operations

MajorNo crash reporting or telemetry

Everything is print() to a console the user will never see once packaged. The circuit breaker and thread supervisor added this session log to stdout only — there's no way to know a real user's session degraded unless they report it themselves.

MinorNo health/readiness distinction exposed to the frontend

The live overlay doesn't surface a "backend unreachable" or "websocket disconnected" state to the user — failures are logged to the browser console only.

Frontend

MajorNo accessibility support

No ARIA roles, no keyboard navigation, no focus trapping in modals — every interactive control is a bare <div onClick> or unstyled button. Not a checkbox item for a genuinely shippable desktop app.

MinorStatus polling can stomp in-flight edits

App.tsx polls /api/status every 5s and unconditionally overwrites model/language settings from backend state on every tick — can silently discard a change the user is mid-way through making in Settings.

MinorInconsistent failure UX

Some fetch failures show a blocking alert(), others fail silently to console.error with no user-visible signal at all.

Suggested roadmap

Ordered because each phase genuinely depends on the one before it — there's no value shipping polish on top of a build that doesn't produce a working binary.

0
Stop the bleeding
Nothing else matters if the app can't be built or the git history is a liability.
  • Rebuild and commit a working sidecar binary; write one scripted, repeatable PyInstaller build (a real .spec file, not ad-hoc invocations)
  • Rotate the exposed HF token; scrub .env and the token from git history; confirm .env stays out of future commits
  • Fix or remove the README's "no audio leaves your machine" claim to match reality
1
Make it safe to hand to a second person
Anything beyond "just me, on my machine" starts here.
  • Add a visible recording indicator and a consent/notification step before capture starts — this is the single highest-liability gap in the whole audit
  • Add basic API auth (even a shared local token) and drop the CORS wildcard; set a real Tauri CSP
  • Encrypt saved transcripts/audio at rest; add a retention/deletion mechanism and a way to purge the 328MB (and growing) history folder
2
Make it trustworthy to run unattended
The reliability work already started this session — finish the loop.
  • Stand up a minimal test suite: at minimum, the VAD/gating logic and hallucination-filter behavior that were hand-validated this session, as real pytest cases against fixture audio
  • Add CI that runs those tests and a build check on every push
  • Wire up crash reporting (Sentry's Tauri community plugin, or equivalent) and an auto-updater so fixes can actually reach installed users
3
Close the gap with production competitors
Only worth doing once 0–2 are solid — this is where accuracy/UX parity lives.
  • Streaming partial results, so text appears within the utterance instead of after the 500ms hangover + inference
  • Replace the single global inference lock with per-channel model instances or real request batching, so cross-talk doesn't force one channel to queue behind the other
  • Decide deliberately whether the 2-channel diarization ceiling is acceptable for the target use case, or whether N-speaker (neural) diarization is worth the added model weight for group meetings
  • Basic accessibility pass and consistent error UX in the frontend
Audit scope: full repo scan (backend, frontend, Tauri/Rust shell, build config, test/CI presence) plus external research on how comparable production systems handle diarization, streaming ASR, GPU serving, and compliance.
